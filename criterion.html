A

Criterion A — Planning / Investigating

Why this project exists and what problem it solves.

Problem & Goal

Many students have projects in different places (local files, GitHub, Google Drive) and no single place to present them clearly. The goal of this website is to provide one profile page that shows a student’s name, skills and projects in a clean, shareable way.

Target audience

  • Students who want to organise and present their projects.
  • Teachers who want to see progress and evidence of learning.
  • Potential future schools or clubs who need a quick overview.

Research

Before building the site I looked at portfolio websites and social platforms such as GitHub, Dribbble and LinkedIn. I noticed:

  • Clear profile section at the top (photo, name, bio).
  • Projects grouped in cards with title, short description and link.
  • Navigation that separates public view from editing tools.

Success criteria

The website is successful if a new user can register, log in, add at least one project and share their profile with another person without needing my help.

B

Criterion B — Design / Structure

How the layout, navigation and user experience were planned.

Information architecture

  • Public profile page: shows name, username, skills, projects and contact info.
  • Dashboard: private area to edit profile and projects.
  • Search page: explore users and projects.
  • Login / Register: access control for each student.
  • Criteria page: explains how the project meets A–D.

Layout decisions

I used a card-based layout with a fixed-width content area so that the website stays readable on laptops and large screens. Sections are grouped with clear headings, spacing and subtle borders.

Wireframe / Component thinking

  • Profile header (picture + text) as reusable component.
  • Project card style reused on profile and dashboard.
  • Forms in dashboard follow the same label + input pattern.

User flow

  1. User visits login page and can see criteria or continue as guest.
  2. User registers → profile document is created in Firestore.
  3. User edits profile / projects in the dashboard.
  4. Public profile updates automatically from the same data.
C

Criterion C — Development / Implementation

Technologies, code structure and advanced features.

Front-end implementation

  • Semantic HTML pages: profile, dashboard, login, search, criteria.
  • Shared navigation bar styled via a global CSS file.
  • Responsive cards and grids using modern CSS (flexbox and grid).
  • Smooth scrolling and hover animations for better UX.

JavaScript structure

  • Single script.js loaded as an ES module.
  • Separate functions for rendering profile, projects, skills and contact.
  • Page initialisation based on CSS classes (profile-page, dashboard-page, etc.).

Firebase integration

  • Firebase Authentication with email & password.
  • Cloud Firestore collection users (document per user).
  • Profile + projects saved and loaded from the cloud.
  • Security rules so users can only edit their own document.

Search & guest access

  • Search page loads all user documents from Firestore.
  • Users can be searched by name, username or project text.
  • “Continue without registration” allows exploring projects as a guest.
D

Criterion D — Evaluation / Reflection

What works well, what could be improved, and what I learned.

Strengths

  • Real authentication and database instead of only local storage.
  • Clear separation between public profile and editing dashboard.
  • Re-usable components (cards, forms, sections) and clean design.
  • Search function that works across all users and projects.

Limitations

  • No file upload yet for real profile pictures.
  • Search is not paginated, which could be slow with many users.
  • Design is not fully optimised for phones yet.

Future improvements

  • Add Google sign-in and optional two-factor authentication.
  • Allow users to choose themes or colour schemes.
  • Add tagging and filtering for projects (e.g. “JavaScript”, “Design”).

Skills developed

  • Connecting a front-end website to Firebase Authentication.
  • Storing and reading structured data from Cloud Firestore.
  • Working with a modular JavaScript file that controls many pages.
  • Planning a project using criteria A–D and documenting decisions.
